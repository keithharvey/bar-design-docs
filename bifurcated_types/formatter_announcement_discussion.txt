@Contributors as you may be aware from https://discord.com/channels/549281623154229250/724293659809284137/1493238696546275489, there are big changes coming to the codebase.

## Dev Setup
We will be transitioning to using the [BAR Devtools](https://github.com/beyond-all-reason/BAR-Devtools/) repository to automate the installation and setup of the tools you need to work on the game, engine, Teiserver, and services.

 **We are looking for devs to test migrating to BAR Devtools, *especially those on Windows,* to make the transition for everyone as painless as possible.**

## Game Code
1. **Package Manager:** We will now be using the Lux package manager to manage 3rd party Lua code. This means no more copy-pasting the code to our codebase and manually maintaining it.
2. **Linting:** linting means automatically formatting code. We will now have a single unified style in the codebase.
3. **Spring Split:** The `Spring` table will now be split into three distinct accessors: `SpringShared`, `SpringSynced`, and `SpringUnsynced`.  This will allow detecting errors within the IDE, rather than having to run the game to expose the errors.
4. **Strong Typing:** The code parser in the IDE will now be able to detect a function being called incorrectly as an error, without having to run the game.

See the overall coordinating issue for details of everything:
https://github.com/beyond-all-reason/Beyond-All-Reason/issues/7408

## The Plan
1. Have select group of devs test transitioning to BAR Devtools
2. Have all devs transition over to using the BAR Devtools setup.
3. Coordinate session to merge transformation PRs
5. Re-run migration scripts to regenerate transformation PRs
5. Merge transformation PRs
6. Run transforms on outstanding PRs to fix merge conflicts
7. Proceed as normal

## tl;dr
**So what do *I* have to do?**
1. Install BAR Devtools
2. Wait a bit
3. Run a couple of commands on your local branches

[4/17/2026 6:30 PM] thvle
I think the important distinction here is “supported path” vs “possible path.”

I do want to acknowledge the concern here, especially from Windows contributors. If your current setup works, hearing that the project is moving toward a more standardized environment can understandably feel like your workflow is being pushed aside.

At the same time, this change is coming from a real project need, not from a desire to make contribution harder or to tell people their current way of working is wrong. BAR is growing into something that has to be more maintainable and coordinated over time, especially as it becomes more formal and moves closer to a published Steam release.

At that stage, the project needs a setup that is documented, repeatable, and realistic for maintainers to support. The hard part is that every fully supported workflow adds ongoing cost in docs, troubleshooting, onboarding, and debugging. That gets harder to sustain as the project gets bigger.

So I do think some amount of standardization is necessary. That is different from saying other approaches are invalid, or that contributors on different setups do not matter. It is more that the project probably cannot keep treating every workflow as equally first-class forever. It is also much easier to onboard and support new contributors when there is a clearly defined primary path.

I think the main thing that would help here is being very clear about what is required, what is only recommended, and what remains possible but unsupported or best-effort. A lot of the frustration seems to be coming from uncertainty there, which is fixable with clearer communication.

To be concrete: if you are only working in the game repo, we can clearly document what remains possible on Windows today, what requires WSL, and what is supported on a best-effort basis. If we do not have that clarity yet, we should pause on broad wording like “everyone will switch over” until we do. Thats why we are still in a testing phase.
