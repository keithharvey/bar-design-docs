I put this in the doc, but copy pasting it here for visibility:

Thanks for putting this together, Marek — and for explicitly framing it as a data point for discussion rather than a merge proposal. The PR itself is good work: the SHA pinning, the no-admin repo-local install, the `_run.cmd`/`.sh` symmetry — all clean. Lux is a relatively friendly tool to package this way and you've shown it's doable.

Here's what I'm weighing as I think about whether to extend this pattern to the rest of the toolchain.

**Reproducibility is the property I keep coming back to.** Right now, our dev environment lives in an immutable distrobox container — `just setup::init` builds it from a Containerfile, and if anything drifts a contributor can `distrobox rm` and rebuild in two minutes with their host untouched. Going native means the install steps modify the host (registry entries from the MSI, files under `.tools/`, PATH adjustments). My worry is that two contributors a month in won't have identical environments anymore — the same "works on my machine" failure mode that I think the container is shielding us against. I don't have a clean answer for how to get that property back without a container.

**Scope is the other thing I keep ruminating on.** This PR is ~333 lines for one tool. The rest of what we'd need on Windows is bigger: stylua, emmylua_check, the cross-compile toolchain (mingw-w64 for the Windows engine binary), the codemod runner, language servers, plus whatever we add over the next year. Granted, several of these (the language servers especially) ship as IDE plugins now and don't need a separate CLI install. But the CLI matters for CI, scripting, and pre-commit hooks — and if each tool that does need a CLI install needs its own pinned-URL/SHA/install/update/uninstall block, that's many thousands of lines of careful PowerShell to maintain across versions. Lux is honestly the easy one — a single tool with a plain MSI and tar.gz.

**The container comparison.** Adding a tool today is roughly: `apt install -y X` in `dev.Containerfile`. Every tool gets added the same way; there's no per-platform bootstrap code at all. That's purely a property of having a real package manager inside an image we throw away — and Windows doesn't quite have an equivalent today. `winget` and `scoop` are close-but-not-quite, especially on the throwaway property.

So my honest read: I think this PR is a great existence proof that *one* tool can be done native, but I'd want a strong story for those two properties (reproducibility + scope-cost) before extending the pattern. WSL2 + container is a once-per-machine setup that gives us both for free — *and* it lands every contributor (Linux and Windows) on the same `just bar::*` CLI surface. Honestly that's the part I'm most excited about: a shared, hackable CLI that anyone can extend rather than just consume. Bifurcating the implementation also splits the contributor pool: most folks who'd dig into the tooling are already bash-comfortable, and a parallel PowerShell tree (which you mentioned isn't your strongest area either) would gate them out of half the codebase. That cuts directly against the "hackable CLI anyone can extend" property I just called out. Even if we have to do some _really_ hacky rsync+systemd voodoo in order to squash performance problems, that's still -1000 lines of code and a unified CLI surface. I'd lean toward keeping that as the supported path — while staying open if you see angles I'm missing on either property. 

*(One thing I want to walk back: I've been using the phrase "not possible" in earlier conversations about Windows-native tooling, and your PR does in fact disprove that. What I should have been saying is "not possible **maintainably**" — i.e. using bash post-setup without modifying the host is the part of Devtools that I think makes it plausible to scale. The PoC is a fair pushback on the looser version of my claim.)*
